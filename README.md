## 1. PRD

*Основная роль: SA/PO*  
*Поддержка: AI Engineer, Fullstack*

### Сегменты пользователей:

1. **Студенты вузов и колледжей**: Основные пользователи, которые непосредственно пишут курсовые, дипломные и другие academic работы. Ценят скорость, простоту и гарантированное качество результата.

2. **Преподаватели (проверяющие работы)**: Косвенные пользователи, которые получают работы на проверку. Заинтересованы в грамотно оформленных, уникальных работах с качественными источниками, что упрощает процесс проверки.

### Pain Points (Боли пользователей):

1. Высокие временные затраты: Студенты тратят недели на поиск источников и написание работы.
2. Сложность оформления: Трудности с ручным соблюдением стандартов ГОСТ, APA, MLA.
3. Стресс и риск сорвать сроки: Давление дедлайнов приводит к панике и снижению качества работы.
4. Низкое качество источников: Сложно найти релевантные и авторитетные научные материалы.
5. Страх перед плагиатом: Неуверенность в уникальности самостоятельно написанного текста.

### Use-cases (Сценарии использования):

1. **"Антипаника-режим"**: Студент загружает тему работы за день до дедлайна, сервис автоматически генерирует структурированный черновик с источниками и оформлением.
2. **"Пошаговое написание"**: Студент пишет работу поэтапно (введение, главы, заключение), получая AI-подсказки и проверку стиля для каждого раздела.
3. **"Проверка и оформление"**: Пользователь загружает готовый черновик, сервис проверяет его на уникальность, перефразирует под академический стиль и автоматически оформляет библиографию.
4. **"Поиск источников"**: Студент вводит ключевые слова по теме, сервис находит и предлагает список релевантных научных статей и книг.

### Value Proposition (Ценностное предложение)

Сервис-хаб для студентов, который в разы сокращает время и убирает стресс при написании academic работ за счет автоматической генерации качественного черновика с проверенными источниками и безупречным оформлением по требуемому стандарту.

### North Star Metric

**Время до готового черновика работы ≤ 20 минут.**  
Целевое значение: 20 минут от ввода темы до получения студентом готового к доработке черновика.

## 2. Детализация Use Cases

*Основная роль: Fullstack*  
*Поддержка: SA/PO*

### Детализация Use Cases

### **Happy Path (Основной успешный сценарий)**

1. Пользователь запускает Telegram-бота
2. Пользователь выбирает тип академической работы (дипломная, курсовая)
3. Пользователь отправляет тему работы
4. AI-модуль генерирует содержание работы на основе типа и темы
5. Система отображает сгенерированное содержание пользователю
6. Пользователь подтверждает содержание
7. Система разбивает содержание на пункты и генерирует полный текст для каждого через AI
8. Система объединяет все части в единый документ
9. Пользователь выбирает стиль форматирования (ГОСТ, MLA, APA)
10. Система автоматически форматирует документ по выбранному стандарту
11. Документ успешно проходит проверку на антиплагиат (>70%)
12. Документ успешно проходит проверку на AntiAI-детекцию (≤50%)
13. Система отправляет готовый документ пользователю

**Ценность:**
- Полностью автоматизированный процесс создания академической работы
- Экономия времени и усилий пользователя
- Гарантия соответствия академическим стандартам
- Минимизация рисков обнаружения AI-генерации

---

### **Alternative Flow(1): Пользователь не одобряет сгенерированное содержание**

1. На шаге 6 основного сценария пользователь отклоняет предложенное содержание
2. Пользователь вносит правки и комментарии к содержанию
3. Система отправляет исправленные требования в AI-модуль для перегенерации
4. Процесс возвращается к шагу 4 основного сценария

**Ценность:** Гибкость настройки под индивидуальные требования пользователя

---

### **<span style="color:orange">[WARNING]</span> Alternative Flow(2): Проверка на антиплагиат не пройдена**

1. На шаге 11 основного сценария показатель уникальности ниже 70%
2. Система уведомляет пользователя о необходимости перегенерации содержания
3. Процесс возвращается к шагу 4 основного сценария с учетом предыдущих наработок
4. AI-модуль генерирует альтернативные формулировки и подходы к теме

**Ценность:** Гарантия уникальности конечного результата, соответствие академическим требованиям

---

### **<span style="color:orange">[WARNING]</span> Alternative Flow(3): Проверка на AntiAI не пройдена**

1. На шаге 12 основного сценария вероятность AI-генерации превышает 50%
2. Система анализирует текст и выявляет "маркеры" AI-генерации
3. AI-модуль переписывает проблемные sections с изменением стилистики
4. Процесс возвращается к шагу 11 основного сценария для повторной проверки

**Ценность:** Защита от обнаружения AI-генерации, повышение естественности текста

---

### **Error handling(1): Технический сбой AI-модуля**

1. Система уведомляет пользователя о временной недоступности сервиса
2. Предлагает повторить попытку через 5-10 минут
3. Автоматически регистрирует ошибку для технической поддержки
4. Сохраняет текущий прогресс работы

**Ценность:** Минимизация простоя, сохранение данных пользователя

---

### **Error handling(2): Неподдерживаемый формат документа**

1. При попытке загрузки сторонних документов система проверяет форматы
2. При обнаружении неподдерживаемого формата показывается сообщение с ошибкой
3. Система предлагает список допустимых форматов (.docx, .pdf)
4. Пользователь может конвертировать файл и повторить попытку

**Ценность:** Предотвращение ошибок обработки, стабильность работы системы

---

### **Error handling(3): Превышение лимита генерации**

1. Система отслеживает объем генерируемого текста
2. При приближении к лимиту пользователь получает предупреждение
3. При превышении лимита система предлагает оптимизировать запрос
4. Для больших работ доступно разбиение на несколько сессий

**Ценность:** Контроль нагрузки на систему, равномерное распределение ресурсов

## 4. Матрица RACI

*Основная роль: SA/PO*  
*Поддержка: All team*

### Матрица ответственности (RACI)

| Область ответственности       | SA/PO (Product Owner) | AI Engineer | MLOps       | Fullstack   |
|-------------------------------|-----------------------|-------------|-------------|-------------|
| Стратегия продукта & Vision   | A                     | C           | I           | C           |
| UI/UX дизайн и прототипирование | A                   | I           | I           | R           |
| Разработка AI-моделей и пайплайнов | C               | A           | C           | I           |
| Бэкенд разработка (API, БД)   | I                     | C           | I           | R, A        |
| Фронтенд разработка (Интерфейсы) | I                   | I           | I           | R, A        |
| ML инфраструктура и пайплайны | I                     | C           | R, A        | I           |
| Деплой и DevOps (CI/CD)       | I                     | I           | R, A        | C           |
| Наблюдаемость и логирование (MLOps) | I               | I           | R, A        | I           |
| Тестирование (Unit, Integration) | I                 | I           | I           | R           |
| Написание документации        | C                     | R (для AI)  | R (для MLOps)| R (для UI/API) |
| Приоритизация backlog         | A                     | C           | I           | C           |
| Коммуникация с ментором       | A                     | I           | I           | I           |

### Легенда:

- **R (Responsible)** — выполняет работу.
- **A (Accountable)** — несет ответственность, имеет право veto («подписант»).
- **C (Consulted)** — консультируется, двусторонняя связь.
- **I (Informed)** — информируется, односторонняя связь.

### Процесс принятия решений:

Финальное решение по спорным продуктовым и техническим вопросам за SA/PO.

### Процесс эскалации:

1. Обсуждение в команде → 2. Решение SA/PO → 3. Обращение к ментору.
